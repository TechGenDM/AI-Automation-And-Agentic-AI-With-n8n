## Quick Recap

This is the **big API module** — the full language of APIs, explained using one running analogy: a **restaurant**. This is one of the most important modules in the whole course, since APIs are the backbone of almost every automation, app, and AI tool you'll ever build.

---

## The Big Picture: Restaurant → Tech

| Restaurant | Tech World |
|---|---|
| You (customer) | Client (app/user making a request) |
| Menu | API Documentation |
| Waiter | The API itself (the messenger) |
| Kitchen | API Server (does the actual work) |
| Your order | Request |
| Your food | Response |

**Key insight**: sometimes there's no human at all — one app can call another app directly. Whoever's *asking* is the **client**; whoever's *answering* is the **server**. These roles can flip depending on who's asking whom.

---

## REST APIs & HTTP Methods (the "verbs" of ordering)

REST is just "the standard way apps talk using HTTP." The methods map directly to **CRUD** (Create, Read, Update, Delete):

| Method | CRUD | Restaurant analogy | Changes data? |
|---|---|---|---|
| **GET** | Read | "Just give me the menu" | No |
| **POST** | Create | "Here's my order" — send data to create something new | Yes |
| **PUT** | Update (full) | "Replace my whole order with this new one" | Yes |
| **PATCH** | Update (partial) | "Just change the delivery time" | Yes |
| **DELETE** | Delete | "Cancel my order" | Yes |

**PUT vs PATCH (commonly confused):**
- **PUT** = client sends the *entire* record, even fields that didn't change → server overwrites everything
- **PATCH** = client sends *only* the field(s) that changed → server updates just that part

## Query Parameters vs Headers vs Body

This is the most important table in this module:

| | Query Parameters | Headers | Body |
|---|---|---|---|
| **Purpose** | Filter/modify the request | Extra metadata/context (incl. auth) | The actual data/content |
| **Where it lives** | In the URL, after `?` | Hidden section of the request | Hidden, separate from URL |
| **Visibility** | Visible in URL | Not visible in URL | Not visible in URL |
| **Format** | key=value | key=value | JSON, form-data, raw text, etc. |
| **Restaurant analogy** | "Gluten-free, extra spicy" (options) | Your reservation ID, loyalty card | The actual dish you're ordering |
| **Example** | `?q=NBA` | `Authorization: Bearer <key>` | `{"query": "NBA"}` |

**Real example**: Google expects your search term as a **query parameter** (`?q=NBA`), but Tavily expects it inside the **body** as JSON (`{"query": "NBA"}`). Same concept (searching), different implementation — always check each API's documentation.

## Authentication (proving you're allowed in)

Just like a restaurant might need a reservation, ID, or loyalty card — APIs need proof you're a legitimate user.

Most common pattern:
```
Authorization: Bearer <your-api-key>
```
This usually goes in the **header**. Miss the word "Bearer" or add an extra space → the call fails. Without a key, anyone could rack up charges on your account.

## API Docs vs Swagger

- **API Documentation** = the full menu, explained in detail (endpoints, methods, auth, examples)
- **Swagger** = an *interactive* menu — you can test-fire a request right there in the browser, like tasting a sample before ordering

## Response & Status Codes

**Response** = whatever comes back, good or bad. It always has a predictable **structure**.

**Status codes** tell you what happened, grouped by first digit:

| Code range | Meaning | Restaurant analogy |
|---|---|---|
| **1xx** | Informational | "We got your order, processing..." |
| **2xx** | Success ✅ | "Dish served!" (200 = the good one) |
| **3xx** | Redirect | "Please go to counter 2 instead" |
| **4xx** | Client error (you messed up) | You ordered wrong, or forgot ID |
| **5xx** | Server error (their mistake) | "Kitchen's on fire" — not your fault |

Codes worth memorizing:
- **200** — OK, all good
- **400** — Bad Request (malformed data)
- **401** — Unauthorized (bad/missing key)
- **403** — Forbidden (authenticated, but not allowed to see this specific thing)
- **404** — Not Found (wrong endpoint)
- **429** — Too Many Requests (rate limited)
- **500** — Internal Server Error (their problem, not yours)

---

## Error Handling, Rate Limiting & Pagination

- **Graceful error handling**: your app shouldn't crash when a call fails — it should catch the error and respond sensibly
- **Rate limiting**: APIs cap how many requests you can make per minute/hour — prevents abuse
- **Pagination**: instead of sending 1 million results at once, data comes in "pages" — much lighter on both server and client

**Bonus terms:**
- **Webhook**: instead of *you* repeatedly asking "is my order ready?", the server pushes data to you the moment something happens (instead of polling). This becomes very important for triggers.
- **Versioning**: APIs release new versions (`v1`, `v2`) while sometimes still supporting the old one.

## Curl (the raw truth of an API call)

**Curl** = a command-line way to fire off any API request directly. Postman and browsers are just pretty wrappers around the same underlying request.

## Full Summary Table

| Concept | Simple definition |
|---|---|
| API | A way for two systems to talk — request in, response out |
| REST | Standard way of using HTTP methods to talk to APIs |
| GET/POST/PUT/PATCH/DELETE | Read/Create/Full-update/Partial-update/Remove |
| Endpoint | The specific address you're calling |
| Query Parameter | Filters, visible in the URL |
| Header | Metadata + auth, hidden |
| Body | The actual data payload, hidden, flexible format |
| Auth | Proof you're allowed to use the service |
| Response | What comes back |
| Status Code | Tells you success/failure and why |
| Curl | Text version of an API call, portable and shareable |

---

## Key Takeaway

- Every API interaction = **Client asks → Server responds** (restaurant order → kitchen → food)
- **GET/POST/PUT/PATCH/DELETE** map directly to Read/Create/Full-Update/Partial-Update/Delete
- **Query params** (visible, in URL) vs **Headers** (hidden, metadata+auth) vs **Body** (hidden, actual data) — always check the docs to know which one an API expects
- **Status codes** tell you exactly what happened — 2xx good, 4xx your mistake, 5xx their mistake
- **Curl** is the universal, portable way to represent any API call — and it's how you import unsupported APIs into n8n

# Module 17: Multi Agent Architecture - Parallel Flow

> Status: ✅ Complete

---

## Quick Recap

This is the **second multi-agent architecture pattern** — the direct opposite of Chain of Agents. Instead of a relay (one after another), **Parallel Agents** run **all at the same time** on the same input, and their results get combined at the end. Think of it as a "team brainstorm" instead of a relay race.

---

## Core Concept

```
Trigger → [Agent A, Agent B, Agent C — all running simultaneously] → Merge → Aggregate → Output
```

Each agent:
- Works **independently**, on the **same input**
- Can use a **different perspective, different tool, or different AI model**
- Doesn't wait for the others — everyone starts and works at the same time

---

## The Intuition (Group Brainstorm Analogy)

> Imagine asking a group: **"What do you think about remote work?"**
> - One person says it boosts productivity
> - Another says it hurts team bonding
> - A third gives a balanced take
>
> Nobody waits for the others to answer first — everyone thinks **at the same time**, and you combine the views afterward for a fuller picture.

This is exactly how Parallel Agents behave — same input goes to everyone at once, and the outputs get merged into one complete result.

---

## When to Use Parallel vs. Chain of Agents (the key decision)

| Use **Parallel** when... | Use **Chain** when... |
|---|---|
| Tasks are independent and order doesn't matter | Each step genuinely depends on the previous one's output |
| Example: researching the same topic from different sources (Wikipedia, Google, etc.) | Example: research → write → review (each step needs the previous one) |
| Example: generating multiple content variations for A/B testing | Example: a chain of approvals |
| Speed matters — everything runs at once | Quality/refinement through stages matters more than speed |

### A subtle but important warning
In a **sequential chain**, one agent's output can unintentionally bias the next agent. Example: if Agent A's research leans positive on a topic, and that colors Agent B's take when it should have been completely independent — that's contamination you don't want.

> **Running agents in parallel avoids this bias problem entirely** — each agent only ever sees the original input, never a peer's opinion or slant. This is a genuine advantage of parallel over sequential, not just a speed trick.

## Building It in n8n — Two New Technical Pieces

### 1. Merge Node
Combines the outputs from multiple parallel branches into one place.
- You configure **"number of inputs"** (e.g., 3, if you have 3 parallel agents)
- Wire each parallel agent's output into this one Merge node
- The Merge node has different modes, e.g.:
  - **Append** — stacks all items into a single list (most common)
  - **Combine / SQL Query** — write a query-style merge for more complex logic
  - **Choose Branch** — output data from just one specific branch without modifying it

### 2. Aggregate Node
Takes the merged list and consolidates it into a single, clean final field.
- Example: you can rename the merged output field to something clear, like `"final_research"`
- This is what makes the merged data actually **usable downstream** — instead of staying as separate, scattered items

### Merge vs. Aggregate (don't confuse these two)
| Merge | Aggregate |
|---|---|
| Combines multiple **input branches** into one list | Collapses a **list** into a single clean output |
| Can have multiple parallel inputs feeding into it | Takes one list as input |
| Used first | Used right after Merge |

> These two are normally used **back-to-back**: Merge first (combine the branches) → Aggregate second (collapse the list into one final usable output).

## Worked Example (built live)

1. **Chat Trigger** — for easy testing
2. **3 copies of an AI Agent**, each connected to a chat model (e.g., all using Gemini, or mixed models)
3. Ask something like: *"What do you think about AI taking jobs? Answer in less than 50 words."*
4. All 3 agents run **at the same time**, each producing a slightly different answer
5. **Merge node** (3 inputs) combines all 3 outputs into one list
6. **Aggregate node** collapses that list into one final field (e.g., `"final_research"`)
7. From here, you can pass this combined result anywhere — send an email, feed it to another agent, etc.

## Stated Benefits

| Benefit | Why it matters |
|---|---|
| **Speed** | Everything runs at once instead of relay-style — total time ≈ the slowest single agent, NOT the sum of all agents |
| **Specialized coverage per task** | Different tools/sources feeding into one complete picture |
| **More balanced/complete results** | Multiple independent viewpoints combined together |
| **Avoids sequential bias contamination** | Each agent only sees the original input — no cross-contamination from a peer's opinion |
| **Scalable** | Adding another parallel agent (e.g., a new API source) doesn't disrupt the rest of the system |

## Practical Note (worth remembering)

When browsing n8n's pre-built templates, many "multi-agent" templates that *look* parallel on the canvas are often actually **Chain of Agents underneath** (just visually laid out differently). This is exactly why understanding the **underlying pattern** — not just how it looks visually — is what lets you correctly identify and build either architecture yourself.

## Key Takeaway

- **Parallel Agents = team brainstorm.** All agents run at the same time, on the same input, then get combined.
- Use parallel when tasks are **independent** (order doesn't matter); use chain when each step genuinely **depends** on the previous one
- **Merge node** (combine branches) + **Aggregate node** (collapse into one final field) are the two new building blocks for this pattern — always used together, in that order
- Real advantage beyond speed: **avoids bias contamination** that can happen in a sequential chain, since every agent only ever sees the original input
- Don't just trust how a template *looks* on canvas — check the actual wiring to know if it's truly parallel or secretly a chain

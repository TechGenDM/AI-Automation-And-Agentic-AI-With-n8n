# Module 20: Multi Agent Architecture - Hierarchical Agent

> Status: ✅ Complete

---

## Quick Recap

This is the **fifth and final architecture pattern** in this series — and it directly resolves a confusion flagged earlier in the Controller/Router module. It's not that "hierarchical" is wrong — it's a **distinct pattern with its own correct use case.** The earlier mistake was using hierarchical when a simple router would've been enough.

---

## Core Concept

```
Trigger → Orchestrator Agent → delegates to Agent 1, Agent 2, Agent 3 (as needed) → back to Orchestrator → Output
```

An **Orchestrator agent** sits at the top. It:
1. Receives the task
2. Understands the context
3. Delegates to specialized **child agents** underneath it

> **Critically — unlike the Router — control RETURNS to the orchestrator after each delegation.** The orchestrator can call one child, get the result back, then decide to call another child, then another — building toward a complete answer, rather than firing-and-forgetting to a single branch.

---

## The Key Distinguishing Feature: Controller/Router vs. Hierarchical

This is the crux of the whole module — the exact confusion from the earlier module, now resolved:

| | Controller/Router | Hierarchical |
|---|---|---|
| **Flow direction** | One-way — task goes out to exactly ONE branch, done | To-and-fro — control keeps returning to the orchestrator |
| **Number of sub-agents per request** | Exactly one | Can call multiple, sequentially, based on what's still needed |
| **Example** | "Route this email to Doubt / Promotion / Technical" | "Create a calendar event AND send an email" — orchestrator calls Calendar agent, gets result back, then calls Gmail agent, then finishes |

> **The rule to remember**: reach for **Router** when a request cleanly belongs to ONE category. Reach for **Hierarchical** when a single request may need coordinated work across MULTIPLE specialized agents, in a sequence the orchestrator itself decides at runtime (not a fixed chain you hardcoded).

### Concrete example that only works as Hierarchical
The Telegram assistant example: *"Can you create a calendar event and send the email as well?"*

This only works correctly under **hierarchical** — because the orchestrator needs to call the Calendar agent, receive confirmation, then call the Gmail agent **with that context**, rather than picking one branch and stopping (which is all a plain Router could do).

## Two Ways to Implement Hierarchical in n8n

### 1. Execute Workflow (sub-agent as a separate workflow)
- Each **child agent lives in its own separate n8n workflow**, triggered via **"When Executed by Another Workflow"**
- The parent (orchestrator) calls it via an **Execute Workflow** node

**Pros:**
- Modular — testable independently
- Reusable across multiple parent workflows/teams — e.g., a shared "Gmail workflow" that 10 different orchestrators can all call
- Best for teams / shared components across an organization

**Legacy reason this existed:** older n8n versions didn't support adding an agent directly as a tool — calling a separate workflow was the only way to achieve this hand-off behavior at all.

### 2. Agent-as-a-Tool (native, newer n8n versions ~1.2+)
- Child agents are added **directly as tools** within the same workflow — each with its own model, tools, and memory, wired in just like any other tool

**Pros:**
- **Shared memory across agents** — single workflow = single visible trace of everything happening
- Better for **real-time / multi-turn assistant-style tasks**
- No need to manage multiple separate workflows
- Supports **nested hierarchy** — a sub-agent can itself have its own sub-agents as tools

---

## Visual Tell: How to Spot Each Type on the Canvas

| Shape | Meaning |
|---|---|
| **Square / Rectangular** node | A plain agent OR a sub-workflow call (Execute Workflow) |
| **Circular / Rounded** node | An agent being used **as a tool** (Agent-as-a-Tool) |

> This is purely a visual distinction in n8n — the underlying behavior can be identical either way. But knowing the shape difference helps you instantly read any workflow diagram and understand which implementation approach was used.

**Important note**: an "Agent as a Tool" node can **only ever function as a sub-agent** — it can't be the top-level orchestrator itself. But this shape supports **multiple levels of hierarchy** — a sub-agent can have its own sub-agents nested underneath it too.

---

## Key Takeaway

- **Hierarchical = an Orchestrator that delegates to multiple child agents, WITH control returning to it each time** — unlike Router, which is a one-way, single-branch decision
- **Rule of thumb**: one category → **Router**. Multiple coordinated steps decided at runtime → **Hierarchical**
- Two implementation methods: **Execute Workflow** (separate workflows, more modular/reusable) vs. **Agent-as-a-Tool** (same workflow, shared memory, better for real-time multi-turn tasks)
- **Visual tell**: square/rectangular = plain agent or sub-workflow call; circular/rounded = agent-as-a-tool
- If you find yourself building a Router but it needs to keep making **follow-up calls** for a multi-step request — that's the signal you actually need **Hierarchical**, not a router with more categories bolted on

## The Full Set: All 5 Multi-Agent Architecture Patterns

| # | Pattern | One-line description |
|---|---|---|
| 1 | **Chain of Agents** | Sequential relay — Agent A → B → C, one after another |
| 2 | **Parallel Agents** | Fan-out — all agents run at once on the same input, then merge |
| 3 | **Controller/Router** | Routes to exactly ONE matching agent/branch |
| 4 | **Feedback Loop (ReAct)** | Draft → Evaluate → Rework, until quality bar is met |
| 5 | **Hierarchical** | Orchestrator delegates to multiple children, control returns each time |

# Module 28: MCP in n8n Cloud

> Status: ✅ Complete

---

## Overview

This module shows how MCP (Model Context Protocol) is exposed and used inside n8n Cloud, and it contrasts cloud behavior with local and self‑hosted deployments.

The lesson focuses on practical setup: MCP server trigger, MCP client, endpoint types, limitations of cloud-hosted MCP support, and debugging tips.

---

## Core Idea

n8n can act as an MCP server host (exposing tools) or as an MCP client (consuming remote tools). In cloud-hosted n8n you get a simplified MCP experience: you can expose tools via an MCP server trigger and connect an MCP client to an endpoint URL, but some transport types (like STDIO) are not available in cloud mode.

---

## What You Can Do in n8n Cloud

- Create an MCP Server Trigger node to expose workflows/tools as MCP endpoints
- Add any n8n-compatible integration (Dropbox, Gmail, Google Maps, custom functions) as a tool on that server
- Retrieve production and test endpoint URLs to connect MCP clients
- Use the MCP Client node to point an agent/workflow to a remote MCP server URL

These features let you decouple tool logic from agents and reuse capabilities across different workflows and agents.

---

## Cloud vs Local / Self-Hosted

Cloud limitations:

- STDIO transport is unavailable in the cloud — you must use HTTP/streamable endpoints
- You can't run arbitrary local MCP servers from the cloud instance; local servers require self-hosting or running the MCP server on your machine
- Cloud MCP support is intentionally limited (expose tools only), so complex local integrations are better handled by self-hosted setups

Local/self-hosted benefits:

- Full STDIO and local process integration for low-latency connections
- Easier to run third-party MCP servers (via Docker, Python, or pnpm scripts)
- True flexibility to host MCP servers that other services can connect to directly

---

## How to Wire MCP in n8n Cloud (Practical Steps)

1. Add an `MCP Server Trigger` node in an n8n workflow.
2. Configure the trigger by selecting the tools/resources you want to expose (Dropbox, Gmail, custom nodes).
3. Save — n8n provides a test URL and a production URL for that MCP server endpoint.
4. In your agent or another workflow, use the `MCP Client` node and paste the provided endpoint URL (HTTP/streamable).
5. Ensure authentication is configured if your MCP server requires API keys or tokens.
6. Run the agent and monitor the MCP Server Trigger logs for tool calls and errors.

---

## Common Errors & Debugging

- "Could not connect to server": check endpoint URL and that the MCP server trigger is active.
- "Authentication attempt to tool failed": ensure credentials (API keys / tokens) are correctly set in n8n and exposed tools.
- Missing STDIO: remember cloud does not support STDIO; switch to HTTP/streamable transport or self-host n8n locally.
- Tool execution errors: inspect the node that implements the tool (input schema, required parameters) and match expected payloads.

Tips:

- Run initial tests in the n8n editor using the test URL before wiring production agents.
- Use clear naming for tools so agents can discover them easily.

---

## Example Workflow Pattern

- `MCP Server Trigger` (exposes: search, sendEmail, listFiles)
- Agent (MCP Client) → calls `search` tool on server
- Agent receives result → calls `sendEmail` tool on same MCP server

This decouples agent logic from specific third‑party API wiring.

---

## Practical Recommendations

- Use n8n Cloud MCP Server Trigger for lightweight sharing of common tools across teams.
- For full MCP capabilities (STDIO, custom local servers), run n8n locally or self‑host it.
- Standardize payload schemas for tools so agents can call them predictably.
- Protect tokens and credentials — MCP endpoints may be accessible if tokens leak.

---

## Key Takeaways

- n8n Cloud supports MCP in a limited but useful way: expose tools via MCP Server Trigger and consume them via MCP Client HTTP endpoints.
- For production-grade, flexible MCP setups (STDIO, local tooling, advanced integrations), self-hosting is recommended.
- MCP in n8n enables modular, reusable tool surfaces that decouple agent logic from API wiring.

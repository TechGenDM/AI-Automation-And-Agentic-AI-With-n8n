## Using SerpAPI

The main tool used in this module is SerpAPI.

SerpAPI is a service that lets you query search engines and travel-related results through an API.

It is useful because:

- it gives programmatic access to search results
- it can fetch flight data and related travel information
- it can be used inside an n8n workflow as a tool

This is a very practical example of how an AI agent can use an external API as a tool.

---

## Why SerpAPI Is Needed

The travel agent needs data that is not normally available in a regular AI model alone.

A model may know general facts, but for live travel information we need current search results.

That is why SerpAPI is useful:

- it acts as a bridge between the AI agent and live travel data
- it does the actual search request for the agent
- the agent then interprets and summarizes the results

## Important Prompting and Agent Design Tips

A key lesson in this module is that the agent’s behavior depends heavily on how it is instructed.

The system prompt should clearly tell the agent:

- it is a travel agent
- it should use travel tools when asked about flights or hotels
- it should use the current date if relevant
- it should avoid guessing when the data is incomplete
- it should provide clear and structured results

This is why good prompting matters a lot.

## Date Handling and Context Awareness

One of the most important practical issues shown in the module is date handling.

The agent may make mistakes if it does not correctly understand:

- the departure date
- the return date
- the current date
- the year

The instructor emphasizes that the assistant should be given the current date and context so it can form correct travel queries.

This is a major insight for real-world agents:

> A model may sound confident, but it still needs clear context to avoid bad assumptions.

## Subworkflow Trigger for Each Agent

A very important architectural pattern shown here is the subworkflow trigger.

Each specialized agent is not just a standalone workflow. It is designed so that it can also be invoked by another workflow.

This is done by adding a trigger that allows the workflow to be executed by another workflow.

Why this matters:

- the agent becomes modular
- it can be reused by other workflows
- it can be called by a master orchestrator

This is exactly the pattern needed for a swarm-style assistant.

---

## Master Workflow Orchestrator

Once the travel agent is built and made callable, the main assistant workflow can use it as a tool.

The master workflow acts like the “brain” of the system.

It receives the user request and decides which specialized agent should handle it.

For example:

- if the user asks about flights, it calls the travel agent
- if the user asks about email, it calls the email agent
- if the user asks about research, it calls the research agent

This is the heart of the swarm approach.

---

## Flow of the Master Assistant

The flow is simple:

1. User sends a request.
2. The main assistant receives the message.
3. It decides which sub-agent is relevant.
4. It calls that sub-agent.
5. The sub-agent performs the task.
6. The result is returned to the master assistant.
7. The master assistant replies to the user.

This is what makes the system feel like a coordinated team rather than one monolithic chatbot.

In a real business setting, this means you can gradually grow the assistant into a more capable system.
